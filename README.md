# HÄFELE Angular App

A small storefront Angular PWA — product listing, product detail, cart and a freely
designed homepage. Product data is a static snapshot of the public FakeStore API
(served from the app's own origin, because Cloudflare blocks SSR requests to
fakestoreapi); the cart checkout still posts to the live API. See the trade-offs for why.

> **Status:** this README started as a decision log and got filled out as the implementation landed.

> **AI usage disclaimer:** I wrote the app code myself and used Claude Code for side-by-side learning (as I usually do). The new functional NgRx store setup was fairly new to me and needed some assistance. Tests and docs are partially generated by Claude Code.

## Overview

The task: a mini Angular PWA with a Product Listing Page (PLP), a Product Detail Page (PDP), a cart and a homepage of my own design — latest LTS Angular with SSR, NgRx/RxJS and basic key-value i18n.

### Links

- **Live demo (Vercel, SSR):** https://haefele.smallstack.app

### Getting started

Prerequisites: Node `20.19+` / `22.12+` / `24+` and npm `11+`.

```bash
npm install               # install dependencies
npm start                 # dev server -> http://localhost:4200
npm run build             # production build
npm run serve:ssr:shop    # run the SSR node server (build first!)
npm test                  # unit tests (Vitest)
npm run e2e               # e2e tests (Playwright)
```

### Zoneless

Runs zoneless (the Angular 21 default, no Zone.js). Change detection comes from signals and the `async` pipe instead of Zone.js patching async APIs. Biggest divergence from Intershop, which still uses Zone.js. The few imperative bits (scroll-to-top on navigation, closing the mobile menu) are written so they don't depend on zone-triggered change detection.

### Standalone components

Everything is standalone, no NgModules. Dependencies are declared per component in `imports`, app-wide providers live in `app.config.ts`. Angular 21 default; differs from Intershop's module setup but keeps deps explicit and per-route lazy loading simple.

### NgRx (functional)

Modern, mostly-functional NgRx:

- `createActionGroup` — typed, grouped actions, named as events (not commands)
- `createFeature` — bundles the reducer + auto selectors; `@ngrx/entity` normalizes products into `{ ids, entities }`
- `createSelector` — memoized selectors, including a cross-slice one that joins cart items with product entities for line totals
- effects are class-based (the one classic bit); `switchMap` for loads (latest wins), `exhaustMap` for checkout so double-clicks don't double-submit
- facades (`ProductsFacade`, `CartFacade`) so components never touch the store directly — keeps them dumb and testable
- a small cache guard skips the product fetch when it's already loaded

### SSR

SSR via `@angular/ssr`. Product data is fetched on the server and ships in the initial HTML; on hydration the client replays the same GET from the HTTP transfer cache (`withHttpTransferCacheOptions`), so there's no double-fetch. Platform guards (`isPlatformBrowser`) keep browser-only code (scroll, dropdown) off the server. Because Cloudflare blocks fakestoreapi from our SSR server, products are served as a static `/products.json` from our own origin (a `baseUrlInterceptor` makes the relative URL absolute during SSR).

### i18n

Basic key-value i18n with ngx-translate. Translations are nested JSON (`en.json`, `de.json`) in `src/app/i18n/`, loaded by a custom loader that bundles them at build time instead of fetching over HTTP — avoids a round-trip and the SSR key-flicker. The initial language isn't hard-coded: `I18nService` negotiates it from the browser's `Accept-Language` header during SSR (English fallback) and transfers it to the client so hydration doesn't flicker; users can then switch at runtime via the language chooser (persisted in localStorage). German visitors get German automatically; the Playwright suite pins `Accept-Language: de` so it's deterministic. Downside: new keys need a rebuild.

### Homepage

The homepage was left open on purpose, so I reused data I already have: it shows the top-rated products plus a link to the full catalog. No extra endpoints, and it warms the product cache so the PLP is instant. It doubles as a soft landing instead of dumping the user straight into the full list.

### Project layout

Feature-oriented, close to how Intershop groups state by domain:

```
src/app/
  core/        # singletons: services, models, interceptors, constants, store (feature folders)
    store/
      products/  # actions, feature (reducer+selectors), effects, selectors, facade
      cart/      # same shape
  shared/      # reusable, mostly-presentational components + pipes
  pages/       # routed feature components (home, products, product-detail, cart)
```

## Assumptions & trade-offs

**All products loaded locally.** The store assumes we only handle a few products. With thousands (each with variants, images, prices) this would be inefficient — you'd want paginated API access (FakeStore doesn't offer it) and a "product readiness" marker like Intershop, so a list can fetch few fields and a PDP more. It's especially bad with SSR + transfer cache, since it bloats the initial payload. Better options: don't load products during SSR (or only a few), show skeletons and hydrate client-side, or add TransferState for the store (small win here — the transfer cache already covers the fetches).

**FakeStore docs are outdated.** Live endpoints differ from the docs (e.g. products have a `rating` object that isn't documented; the cart endpoint returns `{ productId, quantity }`, not full product objects). I followed the data the API actually returns. In production I'd push the backend team to generate OpenAPI from source so docs and API stay in sync.

**FakeStore has no persistence.** POSTed carts return a fake response and the returned id can't be GETed. So the cart service does the request but the real cart is local-only; a reload loses it.

**Tests.** Vitest unit tests cover the products and cart NgRx slices (reducers, selectors, effects); Playwright e2e covers the main flows (PLP, PDP, cart, homepage, not-found, language chooser). Component-level tests are the main thing I'd still add.

## What I'd improve with more time / "in production"

- use the project's branching model, commit conventions and linter
- optimize for mobile devices
- handle loading i18n files differently (on-demand per language)
- avoid classes where I can (e.g. `ProductService`) — functions are tree-shakable
- avoid RxJS, especially Observables for HTTP calls
- treat currency based on the selected language / ICM store
- add breadcrumb navigation
- use image sizes that match the shop dimensions (e.g. wsrv.nl or bunny.net)
- stricter separation of "dumb" vs. store-aware components, for easier component testing

## What took the most time

- finding the best line between Intershop PWA conventions and current Angular + NgRx
- debugging SSR in production: Cloudflare in front of fakestoreapi blocks SSR requests with a bot challenge, so SSR returned an error page. Workaround: a local copy of the products in `public/`.
